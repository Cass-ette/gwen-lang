# Gwen 设计判据

这篇文档不是口号页。它只回答一个问题：

新语法、新规则、新标准库能力，什么时候该进 Gwen，什么时候不该进。

## 先问这 4 个问题

每个候选设计先过这四关：

1. 它有没有把隐藏上下文变成表面语法？
2. 它会不会制造两套长期并存的主写法？
3. 它是不是让 checker / compiler 更容易判断，而不是更难？
4. 它在真实代码里，是否比补丁式写法更短、更稳？

如果前两条过不了，通常就不该进。

如果第三条过不了，通常说明它只对解释器方便。

如果第四条过不了，通常说明它只是一个“看起来省字”的语法糖。

## Gwen 偏好的东西

- 显式目标
- 单一主写法
- 可审计的控制流
- 明确的作用域边界
- 让错误和副作用写在表面上

Gwen 愿意接受“局部多写一点”，换来下面这些东西：

- 重构时不漂移
- 嵌套时不混乱
- checker 能解释
- compiler 不用猜

## Gwen 警惕的东西

- 只是少打几个字，但把语义藏起来
- 同一个能力给两三种同义写法
- 需要靠习惯、默认规则、上下文脑补才能读懂
- 嵌套一深就需要布尔补丁变量、临时标记、额外注释
- 解释器能做，checker 和 compiler 却很难给出稳定语义

## 点号规则

Gwen 不反对 `.`，但不接受把 `.` 变成万能入口。

当前 `A.B` 主要承担这几种含义：

- 模块命名空间：`http.listen`、`json.stringify`
- 用户对象方法：`order.describe()`
- 少量稳定成员访问

Gwen 不鼓励下面这种双轨长期并存：

- `http.path(req)` 和 `req.path` 同时都算主写法
- `module.name(...)` 与实例属性皮肤来回切换
- 链式调用把副作用、状态变化、隐式转换串在一行里

点号表面默认遵守这几条：

- 模块能力优先写成 `module.name`
- 用户对象行为可以写成 `obj.method()`
- runtime handle 默认优先保留显式 helper，而不是大量属性皮肤
- 同一个能力不长期维持两套主写法

## 以循环控制为例

Gwen 选择命名循环配合 `leave <name>` / `next <name>`，而不是裸 `break` / `continue`。

```gwen
while running do scan
  if bad_line then
    next scan
  endif

  if done then
    leave scan
  endif
endwhile scan
```

这里的取舍很明确：

- 控制目标写在表面
- 多重循环时不用数层级
- 不需要 `done`、`should_break` 这类补丁变量
- checker 和 compiler 能直接验证目标是否合法

单层短循环里它可能更长；但 Gwen 不是按“最短一行”优化，而是按“中型代码还能不能读”优化。

## 新设计进入语言前，至少要有

- 一个真实例子，证明它减少了补丁式代码
- 一个反例，证明它没有把旧代码读法搞乱
- checker 能给出清楚错误
- compiler / runtime 不需要为它维护第二套特殊规则

如果做不到，先不要进语言。

## 一句话标准

Gwen 的标准不是“像不像别的语言”，而是“意图有没有直接写在表面上”。
